# DSIL SDK

## 定位与背景

本文档旨在阐明 Atlas 硬件的底层能力如何转化为可用的软件行为。

当 Atlas 在硬件层面确立了精确的时序基准后，DSIL（确定性传感器集成层）便承担起将这些时序信息转化为结构化数据、同步时间戳以及系统可观测性的关键角色。

在此，硬件真正演变为可被上层调用的基础设施。

# 确定性传感器集成层 (DSIL) SDK

DSIL SDK 是连接 Atlas 硬件与机器人软件系统的核心软件层。它允许机器人研发团队快速完成 Atlas 的评估与验证，并可靠地将其集成至量产系统中。

Atlas Fusion V2 在硬件层面定义了时序边界，而 DSIL 则负责将这一边界转化为 Linux、ROS2 及机器人软件能直接消费的结构化、时序感知型数据。

## 价值与意义

在许多机器人系统中，即便硬件已物理连接，软件堆栈仍普遍面临以下痛点：

-   **可信的时序可见性缺失**
-   **同步状态透明度不足**
-   **板级健康状态监测缺失**
-   **缺少从硬件事件到 ROS2 的标准化接入路径**

这些空白带来了实际的集成挑战：

-   工程师需花费大量时间逆向解析时间戳
-   传感器融合算法的结果难以验证与信赖
-   系统调试依赖零散的脚本和临时日志
-   每个新机器人项目都重复着相同的底层集成工作

DSIL 的使命，正是系统性地消除这一软件负担。

其最终成效是：
-   **时序变得可观测**
-   **同步状态变得可验证**
-   **系统状态变得可检视**

## 核心价值

DSIL 位于 Atlas 底层数据流与机器人应用层之间。

![Atlas 软件架构](/img/Fig%2020.png)

它为宿主系统提供了一套标准化的数据消费接口，统一处理：
-   具备时序感知能力的元数据
-   多传感器同步状态
-   传感器健康状态
-   板卡供电与运行状态
-   经标准宿主接口传输的结构化数据

这使得 Atlas 从单一的硬件能力，真正转变为了可部署的机器人基础设施。

## DSIL SDK v1.0 —— 公共核心规范

DSIL SDK v1.0 定义了 Atlas 平台面向公众的最终稳定软件边界。

它并非一份宏大的开发路线图，而是 Atlas 如何集成至机器人计算堆栈的、稳定且可供生产环境直接使用的明确规范。

## DSIL SDK v1.0 能力架构

DSIL SDK v1.0 围绕四个核心能力域构建，每一部分都为机器人系统提供了独特的价值：

### 1. 时间相关与对齐引擎

本域定义了 DSIL 的核心时序智能。它将原始传感器数据的到达时刻，转换为与 Atlas 硬件事件严格对齐的时序感知数据。

![Atlas 硬件时间戳透传](/img/Fig%2021.png)

主要包含：
-   时间戳相关与对齐引擎
-   具备同步感知能力的 ROS2 时间戳映射 (`header.stamp`)

核心价值：
-   将非确定性的传感器数据流转换为确定性时序域的数据
-   实现多传感器数据的可靠时间对齐

### 2. 系统可观测性

本域将传感器基础设施的内部状态暴露给上层软件。

主要包含：
-   同步源及同步状态模型
-   传感器健康状态可见性
-   板卡供电状态可见性（取决于具体硬件）

核心价值：
-   使时序、同步状态和系统健康信息对机器人计算堆栈完全透明
-   工程师无需开发定制调试工具即可验证系统运行的正确性

### 3. 数据传输与解码

本域定义了 Atlas 与宿主系统间的通信机制。

主要包含：
-   稳定的 CDC 数据传输通道 (`/dev/ttyACM0`)
-   运行时数据解码

核心价值：
-   提供标准化的、无需额外驱动的通信链路
-   确保向宿主软件交付一致且结构化的数据流

### 4. 开发者工具与平台支持

本域旨在降低集成门槛，支持在机器人环境中的部署与验证。

主要包含：
-   命令行工具 (CLI)
-   ROS2 监控集成
-   完善的消息格式文档
-   标准化的安装流程
-   经测试验证的 Ubuntu 和 Jetson 系列平台部署方案

核心价值：
-   显著缩短集成开发周期，加速“首次成功运行”的实现
-   为评估和部署 Atlas 提供一套完整的参考软件层

## v1.0 版本需达成的目标

DSIL SDK v1.0 必须完成 Atlas 核心价值的构建，使传感器基础设施在标准机器人软件堆栈中实现**可观测**、**可验证**和**易用**。

具体而言，它必须实现：
-   **确定性时序感知** —— 传感器数据与稳定的硬件时序域实现对齐
-   **同步过程透明化** —— 系统能够向软件层报告其时钟源和同步状态
-   **系统可观测性** —— 传感器活动、健康状态、供电状态无需定制工具即可获取
-   **ROS2 开箱即用** —— 所有数据和遥测信息均可通过标准 ROS2 工作流访问

以上定义了一套 Atlas 在真实机器人部署场景中具备可用性的最低完整系统行为。

## 设备接口

Atlas 通过标准 Linux CDC 设备暴露遥测数据：

`/dev/ttyACM0`

无需任何自定义内核模块。

## 产品边界定义

Atlas DSIL 仅包含两个公共软件阶段：

### DSIL SDK v1.0
最终确定的公共核心平台。

### DSIL SDK v0.1
评估版软件（详见“评估套件”部分）。

### v1.0 之后的演进路径
任何超出 v1.0 边界的功能，都将归入 **白标 OEM 试点 DSIL SDK** 范畴。

这包括但不限于：
-   面向特定客户的 Atlas PCB 定制版本
-   定制化传感器组合
-   定制化时序拓扑结构
-   特定部署场景的适配
-   深度集成相关的工程化工作

此种边界划分是经过深思熟虑的，旨在确保公共平台的：
-   **稳定性**
-   **可理解性**
-   **可部署性**

同时将更深层次的定制化工程工作保留给 OEM 试点项目。

## 集成理念

DSIL 的设计原则是**增强**而非**替代**现有机器人软件生态。

-   标准的传感器驱动保持不变
-   DSIL 与它们并行运作
-   时序与系统可见性作为独立层被注入

这种方式避免了干扰现有感知与控制管线的正常运行。

## DSIL 的战略角色

对于机器人研发团队而言，DSIL 从根本上改变了集成模式：将原来 **“每个机器人项目重复进行传感器适配”** 的模式，转变为 **“跨产品线复用的、确定性的传感器基础设施”**。

这正是 Atlas 平台提供的长期核心价值。

## 常见问题解答

### DSIL 是否必须搭配 Atlas 硬件使用？
是的。DSIL 的实现依赖于 Atlas 生成的遥测数据、时序信号和板级可观测信息。

### DSIL 会取代现有的 ROS2 传感器驱动吗？
不会。DSIL 保留标准驱动，并在此基础上增添具备时序感知能力的基础设施。

### DSIL 是否需要我重写感知算法代码？
不需要。我们的目标是增强时序可见性和同步性，不强制要求修改现有感知算法流程。

### DSIL 是否会发布一个包含众多版本的长期路线图？
不会。公共边界的设计非常简洁：
-   **DSIL SDK v1.0** —— 核心平台
-   **DSIL SDK v0.1** —— 评估软件

超出此范围的功能将作为 OEM 特定项目进行处理。
