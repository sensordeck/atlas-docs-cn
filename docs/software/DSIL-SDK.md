# DSIL SDK

## 本章定位

本页旨在阐明 Atlas 硬件层的时序能力如何转化为可用的软件行为——这正是我们战略上必须拥有的 **DSIL 层**。

Atlas 在硬件层面锐化时间建模，建立精确的时序基准。  
**DSIL（确定性传感器集成层）** 承担关键转换角色：将硬件时序转化为**结构化数据、同步时间戳与系统可观测性**。

在此，硬件真正成为基础设施，而 DSIL 是连接硬件与上层应用的**专有控制点**。

## 确定性传感器集成层 (DSIL) SDK

DSIL SDK 是将 Atlas 硬件与机器人软件系统连接的核心软件层。它将异构传感器时序转换为**单一的、可用的时间模型**。

- **Atlas** 定义时序边界  
- **DSIL** 使该时序变得：
  - 一致
  - 可观测
  - 可被机器人软件直接消费

## DSIL 解决的核心问题

在真实机器人系统中，传感器之间**没有统一的时间共识**。

每个传感器可能从不同来源产生时间戳：
- 硬件触发的时序
- 内部设备时钟
- 传输到达时间（USB / 以太网）

这些时间戳**无法直接比较**，导致：
- 传感器融合不稳定
- 数据回放行为不一致
- 调试困难
- 跨项目的重复集成工作

大多数系统通过“打补丁”的方式绕开此问题。  
**DSIL 在系统级彻底解决它。**

## DSIL 的职能

DSIL 执行一个核心功能：

> 将所有传感器数据归一化到**公共的 Atlas 时间域**

具体实现方式：

### 1. 解析传感器时间

- 评估时序信号、模式和行为
- 理解每个传感器表达时间的方式

### 2. 建立与 Atlas 的时间关系

- 将传感器数据与硬件时序事件关联
- 追踪随时间变化的时序行为
- 持续优化对齐

### 3. 生成修正后的系统时间

DSIL 完成如下转换：

原始传感器时间戳 / 到达时间 → **Atlas 对齐时间戳**

这为所有传感器提供了**一致的时间参考**。

### 4. 将时序作为系统数据暴露

DSIL 使时序可观测：
- 时间偏移
- 漂移行为
- 同步状态
- 时序置信度

## DSIL 时序模型

DSIL 将每个传感器流视为一个**时间映射问题**。

每个传感器具有：
- 一个**原生时序域**
- 与 **Atlas 时间**的关系

DSIL 持续维护：

传感器时间 <--映射--> **Atlas 时间**


该映射可以是：
- 直接与硬件时序绑定
- 从可观测行为推断
- 根据传输时序模式估计

## “修正”的含义

DSIL **不修改传感器**。

它不：
- 重写固件
- 覆盖设备时钟
- 替换驱动

而是：
- 传感器正常工作
- Atlas 定义系统时间
- DSIL 将数据对齐到该时间

这保证了：
- 与现有驱动兼容
- 不破坏现有处理管道
- 最小集成摩擦

## DSIL 产出的内容

每个传感器流变成：
- 对齐到公共时间参考
- 可跨传感器比较
- 时序质量可观测

### 示例输出

```yaml
sensor: front_camera
raw_timestamp_ns: 1712241000123456789
atlas_timestamp_ns: 1712241000122356789
time_offset_ns: -1100000
drift_estimate_ns: 42000
sync_state: tracking
confidence: medium
```

## DSIL 实现的能力

通过 DSIL：
- 所有传感器在**共享时间模型**下运行
- 时序差异变得**可见且可测量**
- 融合管道接收**一致的输入数据**
- 数据回放变得**确定且可重复**

最重要的是：

> **时间成为可靠的系统属性，而非隐含假设。**

## 系统可观测性

DSIL 向主机暴露内部系统状态：
- 同步源与同步状态
- 传感器时序行为
- 系统时序健康度
- 板级遥测信息

这使得工程师能够：
- 实时验证同步
- 无需定制工具即可调试时序问题
- 信任跨部署的系统行为

## 设备接口

Atlas 通过标准 Linux CDC 设备暴露 DSIL 遥测数据：

`/dev/ttyACM0`

- **无需**自定义内核模块
- **无需**修改驱动

## 支持的宿主系统（单板计算机要求）

DSIL 运行于标准 Linux 机器人计算平台。

### 最低环境要求

| 组件 | 要求 |
| --- | --- |
| 操作系统 | Ubuntu 22.04 LTS |
| 内核版本 | Linux 5.15+ |
| ROS 版本 | ROS2 Humble |
| 依赖库 | `libusb-1.0-0`, `python3-serial` |

### 支持的平台

- NVIDIA Jetson（Orin Nano、Xavier、Orin NX）
- ARM 单板计算机（RK3588 类）
- x86 Linux 系统

**要求**：任何支持 USB CDC 设备的系统均可运行 DSIL。

## 集成模型

DSIL **完全运行在用户空间**。

它不需要：
- 内核驱动
- 设备树修改
- 实时操作系统补丁

这实现了：
- 快速启动
- 低集成风险
- 与现有堆栈的兼容性

## 集成理念

DSIL 的设计原则是**增强而非替代**现有系统。

- 传感器驱动保持不变
- 感知管道保持完整
- DSIL 与现有软件并行运行

它增加的是：
- 时序一致性
- 系统可见性
- 基础设施能力

## DSIL 的战略角色

**没有 DSIL：**  
每个机器人项目都从零重建传感器时序逻辑

**有了 DSIL：**  
时序成为跨产品复用的基础设施

这使集成模式从：

> 每个机器人的定制工程投入

转变为：

> 可扩展、可重复的系统架构

**这正是 Atlas 平台提供的长期核心价值，也是我们必须拥有 DSIL 层的战略原因。**

## 产品边界

DSIL 具有清晰的公共边界：

- **DSIL SDK v1.0** —— 生产级核心平台
- **DSIL SDK v0.1** —— 评估版软件

超出此范围的功能将归入 **白标 OEM 试点 DSIL SDK**：

- 面向特定客户的 Atlas PCB 定制版本
- 定制化传感器组合
- 定制化时序拓扑结构
- 特定部署场景的适配
- 深度集成相关的工程化工作

此边界划分是经过深思熟虑的，旨在确保公共平台的：
- **稳定性**
- **可理解性**
- **可部署性**

同时将更深层的定制化工程工作保留给 OEM 试点项目。

## DSIL 不做什么

DSIL **不**：
- 替换传感器驱动
- 强制传感器使用单一接口
- 在不存在确定性的地方声称确定性
- 要求定制感知软件

它与现实系统**原样**工作。

## 与 ROS2 的关系

DSIL 运行于 ROS2 应用层之下。

它确保 ROS2 接收到：
- 修正后的时间戳
- 同步状态
- 可观测的时序行为

ROS2 节点保持不变——但它们的数据变得**可信**。

## 总结

DSIL 将 Atlas 硬件时序转化为可用的系统行为。

通过：
- 归一化异构传感器时序
- 将数据对齐到共享时间权威
- 将时序作为可观测的系统数据暴露

DSIL 将同步从：

> 隐藏的、不一致的行为

转变为：

> 确定性的、可检查的系统能力

## 常见问题解答

### DSIL 是否必须搭配 Atlas 硬件使用？

是的。DSIL 的实现依赖于 Atlas 生成的遥测数据、时序信号和板级可观测信息。

### DSIL 会取代现有的 ROS2 传感器驱动吗？

不会。DSIL 保留标准驱动，并在此基础上增添具备时序感知能力的基础设施。

### DSIL 是否需要我重写感知算法代码？

不需要。目标是增强时序可见性和同步性，不强制要求修改现有感知算法流程。

### DSIL 是否会发布一个包含众多版本的长期路线图？

不会。公共边界的设计非常简洁：

- **DSIL SDK v1.0** —— 核心平台
- **DSIL SDK v0.1** —— 评估软件

超出此范围的功能将作为 OEM 特定项目进行处理。
