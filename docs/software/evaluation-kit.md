# Atlas 评估套件设置

Atlas 评估套件允许机器人团队使用真实的传感器组合来验证确定性的多传感器同步。

该评估旨在回答一个问题：

**Atlas 能否减少跨传感器的时间戳漂移并简化传感器集成？**

一旦系统连接好，完整的评估通常在 30 分钟内即可完成。

# 评估套件包含内容

Atlas 评估套件包含一个参考传感器组合，旨在暴露跨多个时序域的真实同步问题。

## 包含的硬件

| 组件 | 角色 |
| :--- | :--- |
| Atlas 参考板 | 传感器聚合和时序骨干 |
| USB UVC 摄像头 (罗技 C270) | 标准主机时间戳摄像头 |
| 以太网激光雷达 (Livox MID-360) | 独立时钟域传感器 |
| IMU (VectorNav VN-100) | 高速惯性传感器 |
| GNSS (u-blox ZED-F9P) | 全球 PPS 参考时钟 |
| GNSS 天线 | PPS 时序参考（包含） |
| 工业级 USB 线缆 | 传感器连接 |
| PPS 时序线缆 | GNSS → Atlas 时序校准 |

该传感器组合特意包含了多个独立的时序域，以便可以清晰地观察到同步的改进。

特意选用罗技 C270 作为标准的 UVC 摄像头（无硬件同步引脚），以展示即使对于通常依赖主机时间戳的传感器，Atlas 如何改善其时序。

# GNSS 时序说明

随附的 u-blox ZED-F9P GNSS 接收器提供一个 PPS 时序参考。

为获得最佳效果，GNSS 天线应放置在：
•   靠近窗户
•   室外
•   或能部分看到天空的位置

但是，如果无法获得 GNSS 锁定，Atlas 也可以在**内部时间权威模式**下运行，从而允许在室内实验室或无直接天空视野的设施内进行评估。

# 套件到达前的准备工作

大多数机器人团队可以在评估套件到达之前准备他们的计算平台。

## 必需的计算平台

任何能够运行 ROS2 的 Linux 单板计算机。

例如：
•   NVIDIA Jetson Orin Nano
•   NVIDIA Jetson Xavier
•   Intel NUC
•   x86 ROS 工作站

**推荐环境：**
Ubuntu 22.04
ROS2 Humble
Python 3.10+

## 必需的网络设置

系统应允许：
•   USB 设备访问
•   用于激光雷达的以太网连接
•   ROS2 主题传输

## 软件依赖项

安装以下 ROS2 驱动：
usb_cam
livox_ros2_driver
vectornav_driver

这些驱动独立于 Atlas 运行。

**Atlas 不修改传感器固件或驱动程序。**

# 30分钟评估流程

## 步骤 1 — 连接传感器

摄像头 → Atlas USB
IMU → Atlas I²C
GNSS → Atlas UART + PPS

Atlas → 通过 USB 连接至单板计算机

激光雷达直接连接至单板计算机以太网口。

## 步骤 2 — 启动 Atlas
```bash
fusion_start 
```

预期输出：

```bash
Fusion detected
PPS locked
Timing engine active
```

## 步骤 3 — 启动传感器驱动
启动 ROS2 驱动：

```bash
ros2 launch livox_ros2_driver
ros2 run usb_cam usb_cam_node
ros2 run vectornav_driver imu_node
```

验证主题：

```
ros2 topic list
```

**示例：**

```
/camera/image
/livox/lidar
/imu/data
```

## 步骤 4 — 基准时序测量
测量跨传感器时间戳漂移。

```bash
dsil_analyze --baseline
```

**示例结果：**

```
摄像头 ↔ 激光雷达 偏移: 12 ms
IMU ↔ 激光雷达 偏移: 8 ms
```

这确认了系统当前存在传感器未同步的问题。

## 步骤 5 — 激活 DSIL 同步

```bash
dsil_sync
```

DSIL 应用一个动态时间戳偏移映射，将每个传感器的到达时间映射到 Atlas 硬件时序边界。

## 步骤 6 — 验证同步

```bash
dsil_sync_check
```

**预期结果：**

```
摄像头 ↔ 激光雷达 偏移: 0.9 ms
IMU ↔ 激光雷达 偏移: 0.5 ms
```

## 步骤 7 — 生成可视化图表

```bash
dsil_plot
```

**输出文件：**

```
offset_before.png
offset_after.png
```

这些图表直观地展示了时间戳对齐的改进。

## 通过标准
当达到以下指标时，评估即视为成功。

| 指标 | 目标 |
| :--- | :--- |
| 跨传感器偏移 | < 1 ms |
| 同步稳定性 | > 30 分钟 |
| 时间基准 | GNSS 校准 |
| 传感器驱动 | 保持不变 |

这表明 Atlas 能够在多种传感器类型之间建立一个**确定性的时序边界**。

## 评估性能报告
评估结束时，工程团队可以生成一页纸的性能报告。

该报告通常包含：
*   基准漂移测量
*   同步后漂移测量
*   同步改进图表
*   系统架构图

**示例结果：**

| 指标 | Atlas 前 | Atlas 后 |
| :--- | :--- | :--- |
| 摄像头 ↔ 激光雷达 | 12 ms | 0.9 ms |
| IMU ↔ 激光雷达 | 8 ms | 0.5 ms |

这份报告使工程负责人能够在内部提出清晰的建议。

## CTO 推荐摘要
典型的内部推荐结构：

**问题**
多传感器感知堆栈存在时间戳漂移和集成复杂的问题。

**Atlas 结果**
在摄像头、激光雷达和 IMU 之间建立了确定性的时序边界。

**影响**
*   改善了 SLAM 的传感器对齐
*   减少了感知管道的不确定性
*   简化了集成架构

**建议：**
推进至 OEM 集成试点评估。

## 评估后的步骤
如果评估显示出价值，团队通常会进入**白标 OEM 集成试点**阶段。

该试点阶段旨在将 Atlas 适配到量产机器人平台。

### OEM 试点计划包括
*   定制连接器配置
*   传感器接口映射
*   电源分配设计
*   机械安装集成
*   软件集成支持

该试点计划将产出一个为机器人平台定制的白标 Atlas 参考设计。

## OEM 部署验证
在 OEM 试点阶段，将在真实的机器人系统内进行第二次验证。

**验证的指标：**
*   完整机器人传感器同步
*   感知管道稳定性
*   系统延迟影响
*   部署可靠性
*   功耗与热行为

验证通过后，Atlas 将成为机器人量产传感器基础设施的一部分。

## OEM 集成路径
Atlas 旨在直接从评估过渡到 OEM 部署。

您可以通过三种方式将 Atlas 集成到您的产品中：

-   使用 Atlas 作为参考集成平台

-   通过白标 (OEM) 集成计划采用 Atlas

Atlas 消除了重新设计以下系统的需求：

-   传感器布线架构
-   时序同步系统
-   板载电源分配
