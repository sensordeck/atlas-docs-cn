# 传感器同步 | Atlas 文档

## 本章定位

本章阐释为何传感器同步是机器人系统中的基础性瓶颈，以及Atlas如何依据2026年硬件同步平台趋势，将其重构为可观测的系统基础设施。

在理解Atlas之前，必须认清一个事实：**独立传感器时钟域之间的隐性漂移，是导致系统行为非确定性的根本原因**。

这是Atlas架构的起点，也是我们定义DSIL层的战略支点。

### 从独立传感器时钟 → 统一时序结构

现代机器人系统同时依赖多种感知传感器：

*   摄像头
*   激光雷达
*   惯性测量单元 (IMU)
*   全球导航卫星系统 (GNSS)
*   毫米波雷达

每个传感器都基于其**内部时钟**独立运行，彼此之间没有天生的时间关联。

为了让感知、定位、SLAM等管线能够确定性地工作，所有传感器数据必须在**同一时间基准**下被解析。缺少这一基准，系统将遭受难以察觉、更难调试的**隐性时序错误**——这是现场故障中最常见也最隐蔽的来源。

Atlas 引入**确定性时序基础设施**，将传感器时序从隐含难题转变为**可控、可观测的系统属性**。

## 传感器同步为何至关重要

没有统一同步，传感器间的时间戳会发生漂移与错位，直接导致：

*   图像帧与激光雷达点云在时间上错位
*   IMU读数与视觉帧之间存在未知的时间偏移
*   GNSS时间戳与定位管道不一致

最终表现为：

*   定位漂移
*   传感器融合发散
*   数据回放不可重复
*   系统行为非确定性

在绝大多数系统中，这些问题不可直接观测——它们仅以下游感知不稳定、偶发抖动等形式出现。

**Atlas 通过在传感器域建立单一时间权威，从根本上解决此问题。**

## Atlas 传感器同步：硬件成为基础设施

Atlas 传感器同步是一个横跨硬件与软件的系统级时序基础设施。

它**不是**：

*   一个简单的触发发生器
*   一个仅打时间戳的工具
*   传感器驱动的替代品

它提供的是：

*   **硬件定义的时间基准**
*   **确定性的时序边界**
*   **用于关联与对齐的软件时序结构**

### 战略定位：DSIL 层

根据2026年硬件同步平台趋势，Atlas 必须在硬件层面**锐化时间建模**。具体而言：

1.  **Atlas 在硬件层建立时序** —— 精确捕获、分发、锁定时间事件。
2.  **DSIL 将硬件时序转化为结构化数据** —— 包括同步时间戳、偏移模型、系统可观测性。

这正是**硬件成为基础设施**的含义。没有任何传感器厂商、没有任何应用层玩家愿意拥有这一层“传感器管道”。**我们战略上必须拥有DSIL层**。

DSIL 是所有传感器数据进入统一时间域的**必经关口**，也是我们定义机器人时间标准的控制点。

## 系统模型

![Atlas 系统模型](/img/Fig%2022.png)

Atlas 在传感域与计算域之间引入清晰的分离：

*   Atlas 硬件**定义并分发时间**
*   DSIL SDK **关联并对齐**所有输入数据
*   主机系统（ROS2、SBC等）接收到的已是**时间一致**的数据

这彻底消除了每个传感器或驱动独立解决同步问题的碎片化工程。

## 时序集成的现实挑战

并非所有传感器都支持确定性时序控制。Atlas 专为异构传感器组合的现实环境设计。

### 时序集成等级

*   **硬件触发型传感器**  
    通过PPS或触发信号实现确定性时序  
    → 在Atlas时间权威下实现**精确采集对齐**

*   **信号可访问型传感器**  
    通过硬件事件捕获（数据就绪、同步线）对齐  
    → **高置信度关联**

*   **基于USB的传感器**（如UVC摄像头、深度相机）  
    内部时钟与USB延迟导致无法直接时序控制  
    → DSIL SDK 执行**到达时间关联与偏移建模**

*   **基于网络的传感器**（如以太网激光雷达、PTP设备）  
    存在独立或分布式时间域  
    → DSIL SDK 执行**跨域关联与对齐**

Atlas 确保所有传感器——无论是否支持硬件触发——都能参与统一的系统时间模型。

## 同步的实际工作原理

Atlas 同步通过**硬件捕获 + 软件对齐**共同完成。

### 步骤1 — 硬件时序边界（Atlas）

Atlas 硬件层：

*   摄入外部或内部时间基准
*   分发 PPS / SYNC / TRIGGER 信号
*   精确捕获硬件时序事件

此为所有传感器建立**共享的物理时序基准**。

### 步骤2 — 时间戳关联与对齐（DSIL SDK）

DSIL SDK 构建**统一的时序结构**：

*   关联跨传感器数据流的时间戳
*   将传感器数据对齐到 Atlas 时间权威
*   规范化异构时序域

### 同步机制：动态偏移模型

DSIL 应用**动态偏移模型**：

原始传感器到达时间 → 映射到 → Atlas 硬件捕获的时序事件

这是：

*   **系统级的修正**，而非对传感器固件或时钟的修改
*   保证与UVC摄像头、激光雷达驱动、串行传感器、现有ROS2管道的兼容性

## 实现的效果

通过 Atlas 同步：

*   **硬件触发传感器** —— 完全确定性
*   **非确定性传感器** —— 可测量、可修正
*   **所有传感器数据** —— 时间对齐、可比

最重要的是：

**传感器时序从隐藏行为转变为可观测的基础设施。**

## Atlas 时序架构

Atlas 采用**硬件 + 软件**协同的时序系统。

![Atlas 时序架构](/img/Fig%2016.png)

### 硬件层（Atlas）

*   摄入时序源（GNSS PPS、系统时钟等）
*   分发时序信号：PPS_OUT、SYNC_OUT、TRIGGER_OUT
*   建立确定性采集时序边界

### 软件层（DSIL SDK）

*   关联所有传感器时间戳
*   将数据对齐到统一时间域
*   将时序作为可观测的系统数据暴露

两层共同构成完整的时序基础设施。

## 硬件时序边界

Atlas 引入清晰的边界：

*   传感器运行在其**原生时序域**
*   Atlas **确定性地捕获**时序事件
*   DSIL 在数据到达计算单元前**完成所有对齐**

确保：**所有下游感知处理都基于一致的时间**。

## 直连计算型传感器

部分传感器绕过 Atlas 数据路径（如以太网激光雷达、GMSL摄像头、MIPI/CSI摄像头），但仍**参与同步**。

![Atlas 原生时序域](/img/Fig%2018.png)

*   采集行为遵循 Atlas 时序信号或时序关系
*   数据直接流向计算单元
*   DSIL 在软件中**对齐时间戳**

## Atlas 作为时间权威

Atlas 充当感知系统的**单一时间基准**。

![Atlas 单一时间基准](/img/Fig%2017.png)

即使传感器使用不同传输路径，它们共享：

*   **同一个时间基准**
*   **同一个对齐模型**

## 同步精度

| 属性 | 典型值 |
| :--- | :--- |
| 触发信号抖动 | ≤ 1 µs |
| PPS 分发偏差 | ≤ 1 µs |
| DSIL 时间戳对齐精度 | ≤ 1 ms |

*   硬件确保**紧密采集对齐**
*   DSIL 确保**一致的系统时间戳**

## 时序接口

| 接口 | 方向 | 用途 |
| :--- | :--- | :--- |
| PPS_IN | 输入 | 外部时间基准 |
| PPS_OUT | 输出 | PPS 分发 |
| SYNC_OUT | 输出 | 同步信号 |
| TRIGGER_OUT | 输出 | 采集触发信号 |

## 电气与部署考量

Atlas 使用 **3.3V 逻辑电平时序域**。

*   **严禁**直接连接 5V / 12V / 差分系统
*   对于工业接口，必须使用**信号调理电路**

### 设计指导

*   将输出视为**逻辑信号**，而非功率驱动
*   避免未缓冲的**多分支连接**
*   多传感器触发分发时使用**扇出驱动器**

### 信号完整性

同步质量取决于：

*   线缆长度、屏蔽、接地、电磁干扰环境

Atlas 保证其**板级边界的时序**。端到端性能取决于**系统整体设计**。

## 运行模式

### 外部时间权威

GNSS PPS 或外部时钟驱动系统时序。

### 内部时间权威

Atlas 内部生成时序，确保无外部输入时仍能**一致运行**。

Atlas 为多传感器机器人系统建立**确定性时序权威**。
