Atlas 白标 OEM 集成试点计划 | Atlas 文档

本计划专为已完成 Atlas 评估套件阶段并准备进入白标（OEM）集成试点的机器人团队而设。

评估套件在受控环境中验证了 Atlas 的价值。

OEM 集成试点则将这一成果转化为机器人内部面向生产的系统架构。

目标是：

  • 降低集成风险
  • 加速部署就绪
  • 明确通向量产的路径

Atlas 旨在成为一个可跨产品线重用的 DSIL 基础设施层，而非一次性的集成。

计划概述​

Atlas 白标（OEM）集成试点是一项结构化的工程合作，旨在将项目从评估推进到部署就绪状态。

该试点提供：

  • 特定应用的硬件适配
  • 软件集成指导
  • 时序与同步验证
  • 机械与电源系统评审
  • 生产交接规划

参考平台与就绪状态​

Atlas OEM 集成试点基于：

  • DSIL SDK v1.0（用户空间 + ROS2 就绪集成）
  • Fusion V2 参考硬件平台

这些为时序、遥测和集成行为提供了经过验证的基线。

该试点接受 OEM 特定的输入，包括：

  • 传感器组合
  • 计算平台
  • 机械与电源约束
  • 软件架构

在此阶段，Atlas 并非作为固定产品交付。

Fusion V2 平台确立了参考架构，而 OEM 试点则定义了生产配置。

为何选择 Atlas（对比内部开发）​

构建自定义的同步和集成层通常会导致：

  • 零散的时序实现
  • 时间戳对齐不一致
  • 可观测性有限
  • 重复的工程投入
  • 长期的维护负担

Atlas 提供了一个系统级基础设施层，可标准化这些能力，使团队能够专注于自主技术和产品差异化。

Atlas OEM 赋能计划（8 周）​

Atlas 包含一个结构化的赋能计划，以确保 OEM 团队能够独立操作和扩展系统。

阶段划分​

阶段 1 — 系统理解（第 1-2 周）

  • Atlas 架构与时序模型
  • DSIL SDK 基础
  • 集成标准

阶段 2 — 集成与验证（第 3-6 周）

  • IMU / GNSS / UART 集成
  • PPS 验证
  • ROS2 / 原生集成
  • 调试工作流

阶段 3 — 生产就绪（第 7-8 周）

  • 系统验证
  • 性能调优
  • 失效分析

责任边界​

Atlas 不负责传感器驱动或固件。

OEM 拥有设备级驱动。 Atlas 提供同步、集成和可观测性。

软件知识产权与所有权模型​

Atlas 所有​

  • 固件（时序+遥测引擎）
  • DSIL SDK 核心 以二进制（默认）或源代码（可选）形式提供。

OEM 所有​

  • 传感器驱动
  • 应用逻辑
  • 感知堆栈

修改权限​

  • 主机端 → 完全可修改
  • SDK API → 稳定接口
  • 固件 → 可配置，除非另有约定，否则不可修改

Atlas 是可观测和可配置的，不是一个不透明的黑盒。

高层决策FAQ — OEM试点项目决策指南

工程领导团队在决定采用Atlas前需要解答的关键问题​

本节内容旨在解答CTO、工程副总裁及跨职能领导团队在评估是否推进Atlas OEM试点项目时常提出的关键问题。

目标明确：阐明风险、投资回报率、集成影响及生产就绪度。

战略契合度​

Atlas如何融入我们的机器人架构？​

Atlas将成为您计算堆栈底层的传感器基础设施层。

它能实现以下标准化：

  • 传感器连接
  • 电源分配
  • 时间同步

这使得您现有的软件栈（ROS2、感知算法、SLAM）可以保持不变，同时提升系统的可靠性与可扩展性。

这是一个短期工具还是长期平台决策？​

Atlas被设计为一个长期的基础设施层。

在试点阶段采用Atlas，能够实现：

  • 在多个机器人SKU间重复使用
  • 一致的传感器集成架构
  • 减少未来硬件重新设计的周期

风险评估​

集成风险有多高？​

风险低。

Atlas的特点是：

  • 对软件无侵入性
  • 与现有传感器兼容
  • 可伴随当前架构一同部署

试点项目的设计初衷正是为了在生产投入前规避集成风险。

如果Atlas未能达到预期怎么办？​

OEM试点是一个有边界的评估阶段。

可能的结果：

  • 验证集成成功 → 进入生产阶段
  • 发现差距 → 通过定制化解决
  • 方向不匹配 → 无需承诺生产

这确保了在试点阶段不会被强制锁定。

我们长期是否会依赖Atlas？​

Atlas引入了标准化的硬件层，但是：

  • 它不控制您的软件栈
  • 它不替代您的计算平台
  • 它不需要专有传感器固件

与垂直集成解决方案相比，这降低了长期依赖的风险。

工程影响​

我们的团队需要投入多少工程精力？​

试点阶段投入极少。

通常的参与内容包括：

  • 硬件连接
  • ROS2验证
  • 系统级测试

Atlas团队提供支持：

  • 集成指导
  • 系统调优
  • 问题解决

这会打乱我们当前的开发进度吗？​

不会。

Atlas可以与您现有系统并行集成。

您的团队可以：

  • 在不中断当前开发的情况下验证Atlas
  • 直接将测试结果与现有架构进行对比

我们需要重新设计传感器或软件吗？​

不需要。

Atlas兼容：

  • 现有传感器
  • 现有ROS2驱动程序
  • 现有感知处理流程

投资回报率与商业价值​

我们应该预期哪些可量化的收益？​

Atlas能够减少：

  • 集成复杂度
  • 布线和线束成本
  • 工程迭代周期

Atlas能够提升：

  • 系统可靠性
  • 时间同步一致性
  • 跨产品线的可扩展性

Atlas如何降低长期成本？​

没有Atlas：

  • 每个机器人SKU都需要定制集成
  • 传感器布线和供电需要重新设计
  • 同步问题需要反复解决

采用Atlas：

  • 一个标准化的集成模型
  • 跨产品的可复用架构
  • 降低工程和维护开销

为什么不自己内部开发？​

内部开发需要：

  • 多次硬件改版
  • 定制同步架构
  • 固件+软件栈开发
  • 长期维护投入

Atlas提供了一个可立即部署的基础设施层，让您的团队可以专注于：

  • 自主导航
  • 感知算法
  • 产品差异化创新

OEM试点结束时，团队通常会评估：

最终高层问题​

为什么我们现在就应该推进？​

因为延迟采用意味着：

  • 持续存在碎片化的传感器集成
