传感器同步 | Atlas 文档

本章定位​

本章阐释了为何传感器同步是机器人系统中的基础性问题。

在理解 Atlas 之前，关键是要明白：独立的传感器时序为何会导致系统行为的不一致性。

这是 Atlas 架构的起点。

从独立的传感器时钟 → 统一的时序结构​

现代机器人系统依赖多个感知传感器同时运行：

  • 摄像头
  • 激光雷达
  • 惯性测量单元
  • 全球导航卫星系统
  • 毫米波雷达

这些传感器中的每一个都基于其内部时钟独立运行。

为了让感知、定位和同步定位与建图等管道能够一致地运行，所有传感器数据必须在统一的时间基准下进行解析。

若无此统一基准，机器人系统将遭受难以察觉且更难以调试的隐性时序错误。

Atlas 引入了一种确定性时序基础设施，将传感器时序问题从一个隐含的难题，转变为一种可控、可观测的系统属性。

传感器同步为何至关重要​

没有同步，传感器间的时间戳会发生漂移。这将导致：

  • 图像帧与激光雷达扫描点云错位
  • 惯性测量单元读数与感知帧存在时间偏移
  • 全球导航卫星系统时间戳与定位管道不一致

这些问题最终表现为：

  • 定位漂移
  • 传感器融合不稳定
  • 数据回放结果不一致
  • 系统行为非确定性

在大多数系统中，这些问题并不直接可见——它们仅表现为下游感知系统的不稳定性。

Atlas 通过在传感器域建立一个单一的时序权威来解决此问题。

Atlas 传感器同步是什么​

Atlas 传感器同步是一个横跨硬件和软件的系统级时序基础设施。

它不是：

  • 一个触发器发生器
  • 一个时间戳工具
  • 传感器驱动的替代品

相反，它提供的是：

  • 一个硬件定义的时间基准
  • 一个确定性的时序边界
  • 一个用于对齐和关联的软件时序结构

这使得所有传感器——无论其接口或能力如何——都能融入一个共享的系统时间模型。

系统模型​

Atlas 在传感与计算之间引入了清晰的分离：

  • Atlas 硬件定义并分发时间
  • DSIL SDK关联并对齐所有输入数据
  • 主机系统（如 ROS2、单板计算机）接收到的是已经时间一致的数据

这消除了每个传感器或驱动独立解决同步问题的需求。

时序集成的现实挑战​

并非所有传感器都支持确定性的时序控制。

Atlas 旨在应对现实世界中异构的传感器组合。

时序集成等级​

  • 硬件触发型传感器
    通过 PPS 或触发信号实现确定性时序
    → 在 Atlas 时间权威下实现精确的采集对齐

  • 信号可访问型传感器
    通过硬件事件捕获（数据就绪信号、同步线）实现对齐
    → 高置信度关联

  • 基于 USB 的传感器（如 UVC 摄像头、深度相机）
    因内部时钟和 USB 延迟，无法直接进行时序控制
    → DSIL SDK 执行到达时间关联和偏移建模

  • 基于网络的传感器（如以太网激光雷达、PTP 设备）
    存在独立或分布式的时间域
    → DSIL SDK 执行跨域关联和对齐

Atlas 确保所有传感器都能参与到统一的系统时间模型中，即使在无法直接进行时序控制的情况下也是如此。

同步的实际工作原理​

Atlas 同步通过硬件捕获 + 软件对齐共同完成。

步骤 1 — 硬件时序边界​

Atlas：

  • 摄入外部或内部的时间基准
  • 分发 PPS / SYNC / TRIGGER 信号
  • 精确捕获硬件时序事件

这为所有传感器建立了一个共享的时序基准。

步骤 2 — 时间戳关联与对齐（DSIL SDK）​

DSIL SDK 通过以下方式构建统一的时序结构：

  • 关联跨传感器数据流的时间戳
  • 将传感器数据对齐到 Atlas 时间权威
  • 规范化异构的时序域

同步的机制​

DSIL 应用一个动态偏移模型：

  • 原始传感器到达时间
    → 映射到
  • Atlas 硬件捕获的时序事件

这是：

  • 系统级的修正
  • 非对传感器固件或时钟的修改

这保证了与以下设备的兼容性：

  • UVC 摄像头
  • 激光雷达驱动程序
  • 串行传感器
  • 现有的 ROS2 管道

实现的效果​

通过 Atlas 同步：

  • 硬件触发的传感器 保持完全确定性
  • 非确定性传感器 变得可测量和可修正
  • 所有传感器数据 实现时间对齐和可比性

最重要的是：

传感器时序成为可观测的基础设施，而非隐藏的行为。

Atlas 时序架构​

Atlas 引入了硬件 + 软件相结合的时序系统。

硬件层（Atlas）​

  • 摄入时序源（GNSS PPS、系统时钟等）
  • 分发时序信号：
    • PPS_OUT
    • SYNC_OUT
    • TRIGGER_OUT
  • 建立确定性采集时序

软件层（DSIL SDK）​

  • 关联所有传感器的时间戳
  • 将数据对齐到统一的时间域
  • 将时序作为可观测的系统数据暴露出来

这两层共同构成了一个完整的时序基础设施。

硬件时序边界​

Atlas 引入了一个清晰的边界：

  • 传感器运行在其原生时序域
  • Atlas 确定性地捕获时序事件
  • DSIL 在数据到达计算单元前完成所有数据的对齐

这确保了：

所有下游感知处理都基于一致的时间。

直连计算型传感器​

部分传感器绕过 Atlas 的数据路径：

  • 以太网激光雷达
  • GMSL 摄像头
  • MIPI / CSI 摄像头

这些传感器仍能参与同步。

  • 采集遵循 Atlas 时序信号或时序关系
  • 数据直接流向计算单元
  • DSIL 在软件中对齐时间戳

Atlas 作为时间权威​

Atlas 充当感知系统的单一时间基准。

即使传感器使用不同的传输路径，它们也共享：

  • 同一个时间基准
  • 同一个对齐模型

同步精度​

属性典型值
触发信号抖动不大于 1 µs
PPS 分发偏差不大于 1 µs
DSIL 时间戳对齐精度不大于 1 ms
  • 硬件确保紧密的采集对齐
  • DSIL 确保一致的系统时间戳

时序接口​

接口方向用途
PPS_IN输入外部时间基准
PPS_OUT输出PPS 分发
SYNC_OUT输出同步信号
TRIGGER_OUT输出采集触发信号

电气与部署考量​

Atlas 使用 3.3V 逻辑电平时序域。

  • 切勿直接连接到 5V / 12V / 差分系统
  • 对于工业接口，请使用信号调理电路

设计指导​

  • 将输出视为逻辑信号，而非功率驱动
  • 避免未缓冲的多分支连接
  • 对于多传感器触发分发，使用扇出驱动器

信号完整性​

同步质量取决于：

  • 线缆长度
  • 屏蔽
  • 接地
  • 电磁干扰环境

Atlas 保证其板级边界的时序。

端到端性能取决于系统整体设计。

运行模式​

外部时间权威​

  • GNSS PPS 或外部时钟驱动系统时序

内部时间权威​

  • Atlas 内部生成时序
  • 确保无 PPS 输入时仍能一致运行

实现的能力​

Atlas 将同步转化为基础设施：

  • 可靠的多传感器融合
  • 确定性的感知管道
  • 可重复的系统行为
  • 可观测的时序关系
  • 简化的系统集成与调试

总结​

Atlas 为多传感器机器人系统建立了一个确定性的时序权威。

通过结合：

  • 硬件时序分发
  • 精确的事件捕获
  • 基于 DSIL 的关联与对齐

Atlas 将传感器同步从一个临时性的工程难题，转变为一个可扩展、可观测的系统级能力。
